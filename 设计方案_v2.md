## 一、产品定位与价值主张

### 1.1 产品定位

**一句话：** 你的 7x24 私人搭配师 + 智能买前参谋

| 维度 | 说明 |
|------|------|
| 产品名称 | 智搭配 / AI Stylist（暂定） |
| 目标用户 | 22-40 岁，关注穿搭但不精通搭配的都市人群 |
| 核心痛点 | 满柜子衣服却觉得"没衣服穿"；冲动买了新衣，回家发现没裤子配，压箱底 |
| 核心价值 | 少买错、多穿对、告别"没衣服穿" |
| 差异化定位 | 穿搭推荐基于身材体型 + 买前防撞相似度检测 + 脸型体型社交推荐 |
| 商业模式 | CPS 分佣（主要）+ 高级订阅 + 品牌合作 |

### 1.2 用户价值分层

| 用户层级 | 特征 | 核心需求 | 对应功能 |
|----------|------|----------|----------|
| 入门用户 | 新用户，衣橱未建立 | 快速录入衣橱，获得首套推荐 | 拍照录入 + 快速打标 + 天气推荐 |
| 活跃用户 | 衣橱已建立，日常使用 | 每日穿搭推荐 + 购物决策辅助 | AI 穿搭推荐 + 买前防撞 |
| 深度用户 | 已完善个人信息，追求品质 | 精准身材适配 + 社媒灵感 | 脸型体型匹配 + 社媒推荐 |
| VIP 用户 | 付费订阅 | 专属服务 + 真人穿搭师 | 穿搭师服务 + 无广告 |

### 1.3 设计原则

| 原则 | 说明 |
|------|------|
| 身材优先 | 穿搭推荐中身材适配权重最高（50%） |
| 数据本地化 | 用户身材、衣橱数据优先本地存储，保护隐私 |
| 渐进式引导 | 新用户 5 分钟内获得首套推荐，逐步完善数据 |
| 容错设计 | AI 识别不准时提供一键修正，不阻断用户流程 |
| 轻量启动 | MVP 阶段无需真人穿搭师，AI 规则引擎先行 |

## 二、产品架构总览

### 2.1 产品架构图

```
[用户层]
  iOS App | Android App | 微信小程序 | 管理后台
       |
       v
[接入层] API Gateway - 认证/限流/日志/路由
       |
       +-- [用户服务域] User Domain - 用户管理/身材档案/偏好设置
       +-- [衣橱服务域] Wardrobe Domain - 衣物CRUD/自动打标/批量导入
       +-- [推荐服务域] Rec Domain - 穿搭生成引擎/相似度计算/社交分析
       +-- [购物服务域] Shop Domain - 买前防撞/愿望清单/电商对接
       +-- [内容服务域] Content Domain - 社媒爬取/穿搭模板/穿搭师资源
       +-- [系统服务域] Sys Domain - 天气服务/推送通知/数据分析
       |
       v
[AI 引擎层]
  视觉识别(CLIP/SAM) | 向量引擎(Embedding) | LLM推理 | 规则引擎(Expert)
       |
       v
[数据层]
  PostgreSQL | Redis | MinIO | Qdrant
```

### 2.2 模块依赖关系

```
模块 A（数字衣橱）  <-- 基础层
    |
    +---> 模块 B（AI 穿搭推荐）  -- 依赖：衣橱数据 + 身材数据 + 天气
    |
    +---> 模块 C（买前防撞）     -- 依赖：衣橱数据 + 身材数据
    |
    +---> 模块 D（脸型体型+社媒） -- 依赖：身材数据（进阶）
```

**依赖说明：**
- 模块 A 是基础层，必须先完成
- 模块 B 是核心层，依赖 A 的数据
- 模块 C 和 D 是独立增值层，依赖 A 和身材数据
- 模块 D 需要用户完善更多个人信息，属于进阶功能

## 三、模块 A：数字衣橱（基础层）

### 3.1 功能规格

| 功能 ID | 功能名称 | 优先级 | 功能描述 |
|---------|----------|--------|----------|
| A-01 | 拍照录入 | P0 | 调用摄像头拍照，支持裁剪，上传至服务器 |
| A-02 | 选图录入 | P0 | 从相册选择照片，支持批量（最多 30 张/次） |
| A-03 | AI 自动打标 | P0 | 自动识别：品类、颜色、图案、材质、风格、版型 |
| A-04 | 背景去除 | P1 | 使用 SAM/rembg 自动去除背景 |
| A-05 | 季节标记 | P0 | 用户选择：春/夏/秋/冬/四季 |
| A-06 | 场合标记 | P0 | 用户选择：通勤/约会/运动/居家/聚会 |
| A-07 | 手动补标签 | P0 | 用户修正 AI 识别结果 |
| A-08 | 衣橱浏览 | P0 | 网格视图 + 分类筛选（品类/颜色/季节/风格） |
| A-09 | 衣橱搜索 | P1 | 按标签搜索（如"蓝色 衬衫 通勤"） |
| A-10 | 批量管理 | P1 | 批量删除/修改标签/移动分类 |
| A-11 | 衣物详情 | P0 | 查看单件衣物详情、穿着记录、搭配历史 |

### 3.2 交互流程

#### 3.2.1 衣物录入流程

```
[开始录入]
    |
    v
[选择方式：拍照 / 选图]
    |
    +-- 拍照 --> [调用相机] --> [拍照确认] --> [继续添加?]
    |
    +-- 选图 --> [相册选择] --> [确认选择] --> [继续添加?]
                                              |
                                [继续添加?] --是--> 回到选择
                                |
                                否
                                |
                                v
                      [AI 自动识别 + 打标]
                                |
                                v
                      [显示识别结果，用户确认/修正]
                                |
                                v
                      [保存到衣橱] --> [完成]
```

#### 3.2.2 衣橱浏览流程

```
[打开衣橱]
    |
    v
[默认：全部衣物网格视图]
    |
    +-- [筛选] --> 按品类 / 颜色 / 季节 / 风格
    |
    +-- [搜索] --> 关键词搜索
    |
    +-- [点击单件] --> [衣物详情页]
                         |
                         +-- 查看标签
                         +-- 修改标签
                         +-- 查看穿着记录
                         +-- 查看搭配历史
```

### 3.3 技术实现方案

| 技术组件 | 选型 | 说明 |
|----------|------|------|
| 图片上传 | 分片上传 + MinIO | 支持大文件断点续传 |
| 背景去除 | RMBG 2.0 / SAM | 轻量模型，移动端可运行 |
| 视觉识别 | CLIP + 微调分类头 | 品类、颜色、风格、材质识别 |
| 向量存储 | Qdrant / Pinecone | 存储衣物 embedding，用于相似度检索 |
| 本地缓存 | SQLite + Realm | 移动端本地缓存，离线可用 |

**AI 识别 Pipeline：**

```
输入：衣物图片
    |
    v
[背景去除] --> 干净衣物图
    |
    v
[CLIP 编码] --> embedding 向量 (512-d)
    |
    v
[分类头推理]
  +-- 品类分类：上衣/裤子/外套/鞋/配饰（多标签）
  +-- 颜色识别：主色 + 辅色（Lab 空间聚类）
  +-- 图案识别：纯色/条纹/格纹/印花
  +-- 材质识别：棉/羊毛/牛仔/丝绸（基于纹理）
  +-- 风格分类：休闲/商务/运动/街头
  +-- 版型识别：修身/标准/宽松/Oversize
    |
    v
输出：结构化标签 JSON
```

### 3.4 数据流

```
用户拍照 --> 客户端压缩 --> 上传服务器 --> AI 识别 pipeline --> 结构化标签
    |                                                      |
    v                                                      v
本地预览 <-- 返回结果 <-- API 响应 <-- 存入数据库 <-- 保存衣物记录
    |
    v
用户确认/修正 --> 更新标签 --> 重新生成 embedding（可选）
```

## 四、模块 B：AI 穿搭推荐（核心层）

### 4.1 功能规格

| 功能 ID | 功能名称 | 优先级 | 功能描述 |
|---------|----------|--------|----------|
| B-01 | 五维输入采集 | P0 | 天气、行程、日期、身材、肤色 |
| B-02 | 自动天气获取 | P0 | 调用天气 API，获取温度/湿度/降雨/风力 |
| B-03 | 行程选择 | P0 | 用户选择：通勤/约会/运动/居家/聚会 |
| B-04 | 身材数据管理 | P0 | 录入/编辑身高/体重/肩宽/腰线/腿型 |
| B-05 | 肤色录入 | P0 | 用户选择或拍照分析 |
| B-06 | 穿搭生成引擎 | P0 | 五维输入 --> 3 套穿搭方案 |
| B-07 | 身材适配说明 | P0 | 每套方案标注身材适配理由 |
| B-08 | 一键换一套 | P0 | 不满意当前方案，快速替换 |
| B-09 | 搭配记录 | P1 | 记录穿了什么 + 用户评分 |
| B-10 | 穿着提醒 | P1 | 定时推送今日穿搭 |
| B-11 | 搭配收藏 | P1 | 收藏喜欢的搭配方案 |

### 4.2 五维输入详解

| 维度 | 来源 | 数据类型 | 示例 |
|------|------|----------|------|
| 天气 | 自动获取 | 温度、湿度、降雨概率、风力、紫外线 | 18-24C，小雨，湿度 65% |
| 行程 | 用户选择 | 枚举值 | 通勤 / 约会 / 运动 / 居家 / 聚会 |
| 日期 | 系统 | 日期 + 时段 | 今日上午 / 明日全天 / 本周摘要 |
| 身材 | 用户录入 | 结构化数据 | 身高 175cm，肩宽 42cm，腰腹有肉 |
| 肤色 | 用户选择/分析 | 枚举值 | 暖调 / 冷调 / 中性 |

**身材五维数据模型：**

```
body_profile = {
  height: 175,              // cm
  weight: 70,               // kg
  shoulder_width: 42,       // cm，肩宽
  waist_position: "high",   // 腰线位置：高/中/低
  leg_type: "straight",     // 腿型：直/弯/粗/细
  body_fat_distribution: "waist", // 胖瘦分布：腰腹/大腿/均匀
  body_shape: "pear",       // 体型：苹果/梨/沙漏/矩形/倒三角
  neck_length: "short",     // 脖子：短/中/长
}
```

### 4.3 穿搭生成算法（三层权重模型）

**权重分配：身材适配 50% > 美学评分 30% > 实用度 20%**

```
输入：weather + occasion + body_profile + wardrobe + skin_tone
    |
    +-- Step 1: 身材适配筛选（权重 50%）
    |     +-- 加载用户身材画像
    |     +-- 应用扬长避短规则库：
    |     |     +-- 肩宽腰细 --> 垫肩/结构感上衣 + 高腰裤/裙
    |     |     +-- 腰腹有肉 --> 垂坠感/A字版型/H型外套
    |     |     +-- 腿粗/不直 --> 直筒裤/A字裙/阔腿裤
    |     |     +-- 腿短 --> 高腰线设计 + 同色系延伸
    |     |     +-- 脖子短 --> V领/大领口
    |     +-- 输出：适配身材的单品候选集 S1
    |
    +-- Step 2: 美学筛选（权重 30%）
    |     +-- 天气过滤（薄厚、面料、防雨/透气）
    |     +-- 场合过滤（风格匹配）
    |     +-- 季节过滤
    |     +-- 输出：S2 = S1 交集 美学条件
    |
    +-- Step 3: 搭配组合生成（权重 20%）
    |     +-- 硬规则：上+下+鞋 必须完整
    |     +-- 颜色和谐度评分（色轮距离）
    |     +-- 风格一致性评分
    |     +-- 多样性惩罚（30 天穿过的组合降权 30%）
    |     +-- 完整度加分（有配饰 +10%）
    |     +-- 输出：Top-N 搭配方案
    |
    +-- Step 4: 输出 3 套方案
          +-- 每套标注：身材适配理由 + 天气适配 + 完整度
          +-- 按总分排序返回
```

**扬长避短规则库（可配置）：**

| 身材特征 | 推荐版型 | 避免版型 | 推荐单品示例 |
|----------|----------|----------|--------------|
| 肩宽腰细 | 垫肩/结构感上衣 + 高腰裤 | 落肩 oversize | 垫肩西装、高腰直筒裤 |
| 腰腹有肉 | 垂坠感/A字/H型外套 | 紧身、低腰 | A字裙、H型风衣、高腰裤 |
| 腿粗/不直 | 直筒裤/A字裙/阔腿裤 | 紧身裤、短裤 | 直筒西裤、阔腿裤 |
| 腿短 | 高腰线 + 同色系延伸 | 低腰、横条纹 | 高腰裤+短款上衣 |
| 脖子短 | V领/大领口 | 高领、小圆领 | V领衬衫、大翻领外套 |
| 上半身胖 | 深色上装 + V领 | 浅色、圆领 | 深色开衫、V领T恤 |
| 小腿粗 | 阔腿裤/长裙/靴子 | 紧身裤、凉鞋 | 阔腿裤、长款半裙 |

### 4.4 输出格式

```
搭配一：通勤商务风（匹配度：92分）
  +-- 上装：浅蓝条纹衬衫（微宽松版型，遮住腰腹）
  +-- 下装：深灰高腰直筒西裤（拉长腿部线条）
  +-- 外套：藏青色休闲西装（垫肩设计，优化肩腰比）
  +-- 鞋：棕色德比鞋（圆头显脚小）
  +-- 配饰：银色手表 + 黑色皮带
  |
  +-- 身材适配：肩宽腰细 --> 垫肩西装强化优势；腿型不直 --> 直筒裤修饰
  +-- 天气适配：18-24C，无雨
  +-- 完整度：4/5（缺袜子）
```

### 4.5 交互流程

  +-- 身材：用户档案（已录入）
  +-- 肤色：用户档案（已录入）

    |

## 五、模块 C：买前防撞参谋

### 5.1 功能规格

| 功能 ID | 功能名称 | 优先级 | 功能描述 |
|---------|----------|--------|----------|
| C-01 | 新衣录入 | P0 | 拍照或截图上传新衣图片 |
| C-02 | 相似度检测 | P0 | 与衣橱中每件衣物计算相似度 |
| C-03 | 相似度分级 | P0 | >90%：几乎一样；70-90%：很相似；<70%：差异较大 |
| C-04 | 身材适配预览 | P0 | 新衣版型 vs 用户身材特征 |
| C-05 | 衣橱融合预览 | P1 | 新衣与衣橱单品逐一虚拟搭配 |
| C-06 | 搭配数量统计 | P1 | 新衣可与衣橱中几件单品搭配 |
| C-07 | 性价比评估 | P1 | 基于搭配数量的购买建议 |
| C-08 | 替代建议 | P2 | 已有同类衣物过多时的多样性建议 |
| C-09 | 愿望清单 | P1 | 收藏待购衣物，跟踪决策 |

### 5.2 交互流程

```
[用户看到喜欢的衣服]
    |
    +-- 拍照 --> [拍摄新衣]
    |
    +-- 截图 --> [从相册选择]
    |
    v
[AI 识别新衣属性]
    |
    v
[相似度检测]
  +-- 与衣橱中每件衣物做 embedding 余弦相似度
  +-- 属性级对比（颜色/品类/风格）
    |
    v
[身材适配评估]
  +-- 新衣版型 vs 用户身材特征
  +-- 输出：适合/一般/不适合 + 理由
    |
    v
[衣橱融合预览]
  +-- 新衣 + 上装候选 --> 评分
  +-- 新衣 + 下装候选 --> 评分
  +-- 新衣 + 鞋/配饰候选 --> 评分
    |
    v
[输出报告]
  +-- 相似衣物：蓝色衬衫（相似度 90%）x1
  +-- 身材适配：修身款不适合腰腹，建议选大一码
  +-- 搭配预览：可与 7 件下装搭配（复用率 70%）
  +-- 购买建议：推荐购买（搭配数充足 + 身材适配）
```

### 5.3 技术实现方案

| 技术组件 | 选型 | 说明 |
|----------|------|------|
| 新衣识别 | CLIP + 分类头 | 识别新衣品类、颜色、风格 |
| 相似度计算 | CLIP embedding 余弦相似度 | 向量距离 + 属性级加权 |
| 身材适配评估 | 规则引擎 + LLM | 版型 vs 身材特征匹配 |
| 搭配预览 | 衣橱查询 + 规则过滤 | 快速筛选可搭配单品 |

**相似度检测算法：**

```
输入：新衣图片
    |
    v
CLIP Embedding --> V_new (512-d)
    |
    v
与衣橱向量库 Qdrant 检索 Top-K
    |
    +-- 余弦相似度 > 0.9 --> "几乎一样，别买了"
    +-- 余弦相似度 0.7-0.9 --> "很相似，建议搭配 XXX 买"
    +-- 余弦相似度 < 0.7 --> "风格差异较大，可以考虑"
    |
    v
属性级加权相似度
  +-- 颜色权重：30%
  +-- 品类权重：25%
  +-- 风格权重：25%
  +-- 版型权重：20%
    |
    v
综合相似度 = 0.6 x 视觉相似度 + 0.4 x 属性相似度
```

### 12.2 详细应对策略

#### R-01：衣物识别准确率低

**风险描述：** AI 自动识别品类、颜色、风格的准确率不足，导致用户体验差。

**应对策略：**
1. **手动修正入口（立即）：** 每件衣物识别后提供一键修正，不强制用户接受
2. **持续迭代模型（短期）：** 收集用户修正数据，持续 fine-tune 模型
3. **冷启动模板库（立即）：** 提供常见衣物的预定义标签，用户选择后快速录入
4. **置信度阈值（短期）：** 低置信度结果自动提示用户确认
5. **多模态融合（中期）：** 结合价格、品牌、购买渠道信息提高准确率

**责任人：** AI 团队
**监控指标：** 识别准确率、用户修正率

#### R-02：用户录入门槛高

**风险描述：** 用户需要录入大量衣物和身材数据，导致流失率高。

**应对策略：**
1. **批量拍照模式（立即）：** 支持一次拍 30 件，后台异步识别
2. **渐进式引导（立即）：** 先录入 5 件即可获得首套推荐
3. **模板快速生成（短期）：** 提供"商务 casual""运动风"等模板，一键套用
4. **游戏化设计（中期）：** 成就系统、连续录入奖励
5. **社交导入（长期）：** 支持导入电商购买记录自动生成衣橱

**责任人：** 产品 + 前端
**监控指标：** 录入完成率、7 日留存率

#### R-03：身材数据敏感

**风险描述：** 用户对身材数据隐私敏感，不愿意提供。

**应对策略：**
1. **本地加密存储（立即）：** 敏感数据本地加密，不强制上传
2. **用户可控删除（立即）：** 随时可删除身材数据
3. **隐私政策透明（立即）：** 明确告知数据用途
4. **匿名化选项（短期）：** 支持匿名使用基础功能
5. **差分隐私（中期）：** 聚合数据分析时使用差分隐私

**责任人：** 安全 + 产品
**监控指标：** 身材数据填写率、删除率

#### R-04：社交平台反爬

**风险描述：** 小红书、微博等平台有反爬机制，导致数据获取不稳定。

**应对策略：**
1. **官方 API（优先）：** 优先使用平台官方开放 API
2. **频率控制（立即）：** 降低爬取频率，模拟人类行为
3. **IP 池（短期）：** 使用代理 IP 池分散请求
4. **缓存结果（立即）：** 爬取结果缓存 24h，减少重复请求
5. **用户众包（中期）：** 用户分享穿搭到 App，获得推荐奖励

**责任人：** 后端 + 合规
**监控指标：** 爬取成功率、封禁率

#### R-05：电商跳转转化低

**风险描述：** 用户通过 App 跳转购买的比例低，CPS 收入不足。

**应对策略：**
1. **优化推荐精准度（短期）：** 提高推荐命中率
2. **专属优惠（立即）：** 与品牌合作提供 App 专属折扣
3. **A/B 测试（短期）：** 测试不同跳转策略
4. **内嵌购买（中期）：** 小程序内直接购买，减少跳转流失
5. **穿搭师推荐（长期）：** 真人穿搭师推荐提高信任度

**责任人：** 产品 + 运营
**监控指标：** 跳转率、购买转化率、CPS 收入

#### R-06：冷启动问题

**风险描述：** 新用户衣橱为空，无法获得个性化推荐。

**应对策略：**
1. **穿搭模板库（立即）：** 提供 100+ 预设穿搭模板
2. **示例数据（立即）：** 新用户初始化 10 件示例衣物
3. **快速录入引导（立即）：** 3 步完成首套推荐
4. **社交导入（短期）：** 导入电商购买记录
5. **朋友推荐（中期）：** 好友间穿搭分享

**责任人：** 产品 + AI
**监控指标：** 首套推荐获取时间、7 日活跃率

#### R-07：AI 推理成本高

**风险描述：** CLIP、LLM 等模型 API 调用成本高，影响盈利。

**应对策略：**
1. **模型蒸馏（短期）：** 将大模型蒸馏为小模型，降低成本
2. **缓存策略（立即）：** 相同输入缓存结果，避免重复推理
3. **边缘计算（中期）：** 轻量模型部署在端侧
4. **批量处理（立即）：** 非实时任务批量处理，降低 API 调用
5. **自托管（长期）：** 核心模型自托管，降低 API 依赖

**责任人：** AI + 架构
**监控指标：** API 调用成本、推理延迟
