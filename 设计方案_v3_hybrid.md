## 一、产品定位与混合架构

### 1.1 产品定位（Hybrid）

**一句话定义：**
> 以 App 为核心体验载体，以小程序为传播与轻量入口，后台统一 AI 引擎与数据服务，形成“核心用户 deepening + 大众用户 onboarding + 穿搭师赋能”的混合产品架构。

| 维度 | 说明 |
|------|------|
| 核心载体 | iOS / Android App |
| 传播/轻量入口 | 微信小程序 |
| 专业赋能端 | 穿搭师助手（Web/小程序） |
| 目标用户 | App 深度用户 + 小程序新用户 + 穿搭师 |
| 核心价值 | 决策代理 + 买前参谋 + 穿搭师服务 |
| 数据层 | 统一云端 + 端侧本地缓存 |

### 1.2 三重端角色
- **App（主阵地）**
  - 承载完整 superpower
  - 负责隐私敏感计算、本地缓存、离线体验
- **小程序（增长引擎）**
  - 降低冷启动门槛
  - 负责传播、分享、轻量推荐、CPS 跳转
- **穿搭师助手（专业端）**
  - 穿搭师查看用户衣橱
  - 提供人工穿搭建议
  - 沉淀穿搭知识库

## 二、端侧定位与分工

### 2.1 功能分配总表

| 能力 | App | 小程序 | 穿搭师助手 | 说明 |
|------|-----|--------|-----------|------|

| 动态标签修正 | ✅ 主 | ❌ | ❌ | 依赖本地反馈闭环 |
| 衣橱浏览 | ✅ 完整 | ✅ 基础 | ✅ 查看 | 图谱浏览器差异化 |
| 穿搭决策中枢 | ✅ 完整 | ✅ 简化 | ❌ | App 支持离线决策 |
| 买前 Wardrobe ROI | ✅ 完整 | ✅ 简化 | ❌ | App 支持本地图库检索 |
| 动态体态档案 | ✅ 完整 | ⚠️ 基础 | ❌ | 照片/隐私优先 App |

## 七、数据流与同步策略

### 7.1 统一数据流
```
App/小程序 -> API Gateway -> 业务服务 -> AI 引擎 -> 数据层
                                    |
                                    v
                             端侧本地缓存/展示
```

### 7.2 同步策略
- **App 优先本地**
  - 衣橱、体态、偏好优先本地
  - 异步同步云端
  - 离线可查看历史决策
- **小程序轻量同步**
  - 不长期驻留本地大图
  - 只保留必要元数据
  - 依赖云端实时计算
- **穿搭师助手授权同步**
  - 用户授权后访问衣橱
  - 只读为主
  - 操作留痕

## 十二、风险矩阵与应对策略

| 风险 | 影响 | 应对 |
|------|------|------|
| App 获客成本高 | 高 | 小程序先行冷启动 |
| 小程序隐私受限 | 中 | 敏感计算放 App |
| 端侧模型一致性 | 中 | 云端兜底，端侧加速 |
| 穿搭师质量不稳定 | 中 | 评分 + 模板化 + 审核 |
| 数据同步冲突 | 中 | 版本向量 + 最终一致 |
| 多端体验不一致 | 中 | 统一设计系统 + 接口层 |

### 商业与成本说明
- 商业链路：CPS 分佣 + 高级订阅 + 品牌合作 + 穿搭师服务
- 成本重点：AI 推理成本、云服务成本、获客成本
- 金额口径：订阅金额、单用户 LTV、ROI 目标
- 性价比校验：以 Wardrobe ROI 和复用率为核心依据，减少高价低效推荐
- 定价边界：避免低价高频推荐占用算力，优先高价值转化场景
